Encoding team standards

Rahul Garg argues that team-specific AI instructions should move from personal prompting habits into versioned repository artifacts. It is worth keeping because it turns AI-assisted coding consistency into an infrastructure problem with a concrete operating model for generation, review, and security workflows.

  • Treat AI instructions like shared config: version them, review them in pull requests, and keep them in the repo.
  • Encode priorities and severity thresholds in the instructions so team judgment executes consistently across different developers.
  • Start with one high-leverage instruction, usually generation or review, then expand only after adoption exposes real gaps.
  • Use incidents, review drift, and repeated corrections as input for updating the instruction set, just like any other engineering infrastructure.

Dependency cooldown using the publish age as a signal for package resolution

PMG's dependency cooldown filters very recent package versions out of registry metadata during version resolution. It is worth keeping because it is a concrete, low-friction supply-chain defense that reduces exposure to freshly published malicious or broken releases without changing normal semver ranges.

  • Cooldown acts at metadata resolution time, so the package manager falls back to older eligible versions automatically when the version range allows it.
  • If only too-new versions satisfy the range, the install fails explicitly instead of silently selecting a risky fresh publish.
  • The control currently depends on proxy mode and covers npm; direct tarball installs and some lockfile or cache-based flows bypass it.
  • A per-command skip flag preserves an emergency escape hatch without removing the default policy for normal installs.
