Fixed-Height Cards: More Fragile Than They Look

The piece shows how fixed card heights, line clamps, absolute-positioned actions, and guessed padding create brittle layouts that fail under translation, missing images, larger fonts, and narrow viewports. The better pattern is intrinsic sizing, grid alignment, flex column flow, and explicit stress tests for real content.

  • A card layout that works only because overflow is hidden is already under strain.
  • Let grid and flexbox handle alignment instead of hard-coding pixel ceilings inside components.
  • Stress-test components with long words, larger text, wrapped controls, missing media, and narrow widths before treating them as stable.

AI likes deep modules

The essay applies Ousterhout's deep-module idea to AI-assisted development: splitting code is not enough if modules leak Kubernetes objects, complex structs, or other implementation details through shallow wrappers. Deep interfaces absorb complexity so both humans and LLMs have less system state to hold.

  • A module is deep when a small interface hides substantial implementation complexity.

  • AI-generated code needs especially clear module boundaries because models tend to reproduce visible surface complexity.

Architecture by Autocomplete

The post argues that AI-generated code often flattens domain concepts into primitive strings, numbers, dates, and maps because that is the statistical average of training data. The review heuristic is concrete: count the primitives in the domain core and replace flattened concepts with domain types and smart constructors.

  • Primitive obsession is a useful diagnostic for AI-assisted code that lacks domain theory.

  • Reviewing AI output should start with types because they reveal whether the domain model was actually understood.

Struggling with flaky end-to-end tests due to data dependencies

The thread surfaces a common test-architecture trap: teams run long end-to-end flows because production data dependencies are hard to model, then every upstream state inconsistency breaks checkout-level confidence. The better direction is deterministic setup, isolated scenario data, and more behavior checks below the full browser/API journey.

  • A checkout test should not need every previous workflow step to succeed just to reach the state under test.
  • Move most behavior checks to API, integration, contract, and component levels; reserve full E2E for a few critical journeys.
  • Use owned seeded data, scenario IDs, reset helpers, and isolated accounts instead of shared mutable QA fixtures.

Experiment Estimation Study: How Well Can 1,391 Guesses Predict A/B Test Effects?

GoodUI compares 1,391 professional guesses against real A/B test outcomes and finds that humans are noisy, mildly optimistic estimators. Crowds reduce random error but do not remove shared positive bias, so estimates are useful for prioritization but cannot replace experiments.

  • The median individual estimate missed the real effect by about 2.7 percentage points.
  • People were directionally right about 64% of the time but tended to overestimate positive impact.
  • Aggregating five to ten estimates reduces noise, but systematic bias remains.
