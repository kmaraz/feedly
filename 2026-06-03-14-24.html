Component Architecture for React Server Components

Shows how React Server Components move data ownership back into server components instead of route loaders, then uses Suspense boundaries and colocated skeletons to design loading behavior explicitly. The durable part is the architecture tradeoff: component autonomy, streaming order, and loading-state drift are treated as design constraints.

  • Route loaders centralize server fetching but couple reusable components to every route that must supply their data shape.
  • RSCs let server components own their reads while React cache handles per-request deduplication, though batching still needs deliberate design.
  • Suspense boundary placement is a UX decision: it controls where users wait and whether sections stream independently or as groups.
  • Keeping skeletons next to the components they mirror reduces layout drift and makes edits easier for humans and coding agents.

The Next.js bloom filter and how it can break App Router migrations

Documents a concrete migration failure where Next.js encodes App Router paths in a client-side bloom filter, false-positives a Pages Router route, forces a hard navigation, duplicates basePath, and produces apparently random 404s. The diagnostic path and config tradeoffs are directly actionable.

  • The client router filter is probabilistic: it avoids false negatives for App Router paths but can false-positive Pages Router paths.
  • The dangerous combination is mixed pages/app routing plus basePath, where a false positive can produce a doubled prefix.
  • Disabling experimental.clientRouterFilter fixes the symptom but removes automatic cross-router hard-navigation detection.
  • Lowering clientRouterFilterAllowedRate trades a larger client bundle for fewer false positives.

Applying The Transactional Outbox Pattern In ColdFusion

Applies the transactional outbox pattern to comment-email delivery in a small ColdFusion app, turning fragile post-transaction external calls into database-backed messages processed asynchronously. The implementation is stack-specific, but the failure analysis is portable.

  • The danger zone is after the database transaction commits but before all required external effects finish.
  • Do not put remote calls inside the database transaction; long and unreliable external work holds locks and connections open.
  • Persisting outbox messages inside the same transaction changes external work from at-most-once toward at-least-once processing.
  • The consumer must handle duplicates, failures, dead-letter behavior, and any hidden async layer such as mail spooling.

The invisible engineering behind Lambda's network

Describes how AWS Lambda's networking team spent years removing VPC cold-start and density bottlenecks by moving expensive tunnel setup off the hot path with eBPF, replacing stateful NAT paths, and drastically simplifying firewall rule scale. It is a strong case study in invisible infrastructure work.

  • The original VPC cold-start cost came from per-function tunnel setup and density-related kernel/network overhead.
  • Precreating tunnels with dummy VNIs and rewriting Geneve headers with eBPF moved setup from the hot path to the pool.
  • The redesign reduced tunnel latency from milliseconds to microseconds and enabled much higher microVM density per worker.
  • The broader lesson is to instrument the full path before optimizing; the bottleneck was not where a simple cold-start model would suggest.
